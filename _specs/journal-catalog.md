## Locked-design constraints that bind every feature below
- **Palette:** deep berry/wine + the `DOM` domain colors (`app.js:249`). NO neon / glow / shine / `.shine` / `.foil` / white surfaces. The pink now-line stays the brightest thing on screen — any new chrome must sit darker.
- **Surfaces** reuse the existing dark-jewel pattern: `mixHex(D.c,"#160510",t)` for fills, `D.light` for text, `D.ink` for on-color text. New sheets/overlays reuse `.bento-ov`/`.bento-card`/`.bento-body` or `#sheet` (`openSheet()`/`closeSheet()`), never a new third menu system (CLAUDE.md landmine — there are already two).
- **One activity at a time.** Don't reintroduce split lanes / concurrent timers.
- **Reward-never-shame** copy. Full-screen, low clutter.
- **Reuse helpers:** `add(p,tag,cls,txt)` `el(id)` `esc(s)` `mixHex(a,b,t)` `fmt(min)` `dur(m)` `hm(t)` `pad(n)` `uid()` `blocks(k)` `logs(k)` `save()` `domainOf(it)` `DOM` `DOM_ORDER` `tiClass(item)` `tiIcon(item)` `bentoPicker(opts)` `allActivities()` `bentoByDomain()` `reflow(k)` `reflowLogs(k)` `renderToday()` `buildPull()` `kd(k)` `keyAdd(k,n)` `key(d)` `relLabel(k)`.
- **HIGH RISK gate:** the day timeline render (`calendarView` `app.js:2274`, the live-bubble / straddle / matched-span logic `~2434–2509`, `buildPull` `app.js:818` + its pinch/swipe gesture wiring `~896–933`) was rebuilt 3×. Anything touching those is HIGH RISK and must be flagged **DEVICE-UNTESTED** in the handoff unless David confirms on his phone. Two of the four features here deliberately AVOID that render path; the notes field touches only the editor sheet, not the timeline.

## Feature 1 — Freeform notes field on logs (and blocks)
**Ask (quote):** *"it will time you in doing that as well as a… a form of digital journal… journaling"* (audit line 710–713, "Doing the plan is itself a form of digital journaling"). Also covers the small ask "Planning can include a journaling session" framing — the executable, concrete piece is: **a freeform notes field attached to each log/block.**

**Buildable?** YES. This is the keystone of the whole cluster — it turns "the log IS the journal" from implicit into a real text field, and it's the data that Feature 2 (journal surface) reads back.

**APPROACH**
1. Add an optional `note` string to log + block objects. No migration needed (absent = `""`).
2. In `editorSheet()` (`app.js:2589`, the ONE unified editor for both logs and blocks via `isLog`), add a notes `<textarea>` block after the length/priority section and before the footer. Auto-save on `input` (debounced) reusing the existing "every change applies live, no Save button" pattern (`commit()` at `app.js:2596`).
3. Surface a tiny note affordance on the timeline bubble WITHOUT touching the render math: append a `ti-note` glyph into the existing `.cn` label `innerHTML` when `o.note` is non-empty. The label is already built at `app.js:2454` (logs) / `app.js:2486` (timers) — add a trailing `(e.note ? ' <i class="ti ti-note" style="color:'+D.light+';opacity:.6"></i>' : '')`. This is additive string concat into an already-rebuilt node, NOT a layout change → low render risk.

**CODE POINTERS**
- `editorSheet(o,k,isLog)` `app.js:2589` — insert the notes UI around `app.js:2654`–`2655` (after STEPS / before MARK DONE), so it appears for BOTH logs and blocks.
- `commit()` `app.js:2596` — already calls `save()`; the textarea handler just writes `o.note` then debounced `save()` (don't call `commit()` on every keystroke — it does `reflowLogs`/`renderToday` and would steal keyboard focus; the architecture landmine. Just `o.note = ta.value; clearTimeout(t); t=setTimeout(save,400);` and a final `save()` on `blur`).
- Label injection: `app.js:2454` (log `cn.innerHTML`) and `app.js:2486` (timer `cn.innerHTML`).

**UI (on locked palette)**
```
  ┌──────────────────────────────────────┐   ← #sheet (.edsheet), dark berry
  │  [🎯 Deep work]            (switch)   │
  │  starts at  − 2:30pm +                │
  │  length  ▁▂▃▅▇  · 2:30–3:30pm         │
  │  ─────────────────────────────────   │
  │  📝 notes                            │   ← ed-hint, color D.light @ .7
  │  ┌──────────────────────────────┐    │
  │  │ felt scattered first 20 min,  │    │   ← textarea: bg mixHex(D.c,#160510,.78),
  │  │ then locked in after coffee   │    │     text #efe3ec, border mixHex(D.c,#160510,.45)
  │  └──────────────────────────────┘    │     placeholder "how did it go? a line is enough"
  │  [✓ Done]                            │
  └──────────────────────────────────────┘
```
Bubble gets a small dim `📝` glyph after its title when a note exists — never bright, never the brightest thing.

**DATA**
- log: `{...existing, note?: string}`; block: `{...existing, note?: string}`. Absent → treat as `""`. **No `load()` migration required** (optional field, old data safe).
- `note` automatically rides through `pushUndo`/`popUndo` (`app.js:1283`, snapshots whole `S.blocks`/`S.log`) and export/import (`app.js:1293`, whole-state envelope). Confirm — nothing extra to do.

**REGION:** `editorSheet` (`app.js:2589–2663`) + two label-string sites inside `calendarView` (`2454`, `2486`). CSS: add `.ed-note` textarea rule in `index.html` near the other `.ed-*` rules.

**EFFORT:** S

**RISKS / regressions**
- Do NOT call `commit()`/`renderToday()` per keystroke — full-DOM-rebuild landmine drops the keyboard (CLAUDE.md). Debounce `save()` only; render on `blur`/Done.
- The two timeline label edits are string-append only — verify the bubble still renders and the `ti-note` glyph doesn't widen thin bubbles into the rail logic (`lbl-i`/`lbl-s` at `app.js:2456`). Glyph is inside `.cn`, which is hidden on degraded/thin bubbles by `degrade()` — so thin bars are unaffected. Spot-check a tall bubble + a thin bubble in preview; gesture feel N/A.

## Feature 2 — Timer tool & journal mirror each other (a journal SURFACE that round-trips)
**Ask (quote):** *"the tool above to time yourself and the journal should be identical… if you add something in the journal, you add something in the tool above"* (audit 690–693).

**Buildable?** PARTIAL → **nearest-concrete-version: a "Journal" reading/writing list-view of the same `logs(k)` the timeline shows, with edits round-tripping through `editorSheet`.** The deeper data model already guarantees they can't diverge (one `S.log`/`S.blocks`); the missing piece David named is a *distinct journal-shaped view* of that same data. We do NOT build a separate journal entity (that would break the "single shared model" the audit says is the whole point) — we build a second *rendering* of the shared model.

**APPROACH**
1. Add `journalSheet(k)`: a `.bento-ov`/`.bento-card` overlay (same chrome as `choresSheet` `app.js:134`) titled "📓 Today's journal" with the day label + a one-line scope nav (‹ Yesterday · Today · Tomorrow ›) reusing `relLabel`/`keyAdd`.
2. Body = a vertical, chronological feed of that day's entries: merge `logs(k)` (what you did) and optionally `blocks(k)` (what you planned), sorted by `hm(time)`. Each row: time · domain glyph (`tiIcon`) · title · `dur(mins)` · and its `note` (Feature 1) shown as a muted sub-line. Each row is a **journal-shaped restatement** of the bubble — proving "the journal and the timer mirror each other."
3. **Round-trip writes both directions:**
   - Tap a row → opens `editorSheet(entry,k,true)` (the SAME editor the timeline bubble opens). Edit title/length/note there → on close, re-render the journal list. → "add/edit in the journal = add/edit in the timer."
   - A "＋ add to today" button at the bottom → `bentoPicker({title:"What did you do?", onPick:fn})` that pushes a new `logs(k)` entry (same write-path as backfill `app.js:2506`). It instantly appears on the timeline next time it renders. → "add in the journal → adds in the tool above."
4. Entry point: add a "📓 Journal" item to `dayToolsMenu` (`app.js:769`, the ⋯ menu — currently Plan day / Enhance / Clear / Undo / Test day) so it lives beside the timeline tools, "above" the day, exactly where David put "the tool above." (Also satisfies the unbuilt audit-134 "Tools menu holds morning/evening journal" partially.)

**CODE POINTERS**
- New fn `journalSheet(k)` — place next to `choresSheet` (`app.js:134`) or near `suggestSheet` (`app.js:2916`); model the overlay scaffold on `choresSheet` exactly (it's the cleanest `.bento-ov` builder).
- Reuse: `logs(k)` `app.js:1321`, `blocks(k)` `app.js:1320`, `editorSheet` `app.js:2589`, `bentoPicker` `app.js:2709`, `tiIcon`/`tiClass` `app.js:301`, `domainOf` `app.js:274`, `relLabel`/`keyAdd` `app.js:101`/`98`.
- Entry point: `dayToolsMenu(anchor)` `app.js:769`.

**UI (on locked palette)**
```
  ┌──────────────────────────────────────┐   ← .bento-card, dark berry
  │  📓 Today's journal     ‹ Today ›  ✕  │
  │  ──────────────────────────────────  │
  │  9:10a  🏃 Run            45m         │   ← row bg mixHex(DOM.move.c,#160510,.74)
  │         "felt strong, cold out"       │     note muted #b7a6b8
  │  10:30a 🎯 Deep work      1h 30m   📝 │   ← tap row → editorSheet (mirrors bubble)
  │  12:00p 🍎 Lunch          30m         │
  │  1:00p  🌫️ Drift          25m         │   ← drift honest, mauve, never hidden
  │  ──────────────────────────────────  │
  │  ＋ add to today                      │   ← bento-go style, opens bentoPicker
  └──────────────────────────────────────┘
```

**DATA:** none new — pure alternate view over `S.log`/`S.blocks`. Reads `note` from Feature 1. No migration.

**REGION:** new standalone fn (no overlap with timeline render) + one line in `dayToolsMenu` (`app.js:769`). Parallel-safe with most other agents EXCEPT another agent also editing `dayToolsMenu`.

**EFFORT:** M

**RISKS / regressions**
- Must NOT create a *separate* journal data store — that would let it diverge, defeating the ask. Read/write only `logs(k)`/`blocks(k)`.
- Editing via `editorSheet` already calls `reflowLogs`/`save`/`renderToday`; after `closeSheet()` re-render the journal list (re-call `journalSheet(k)` or a local `render()`), else it shows stale rows.
- Does NOT touch `calendarView`/`buildPull` render → **not** the HIGH-RISK timeline path. Boots-clean-in-preview is a sufficient claim here; no gesture surface.

## Feature 3 — Reuse the same habit catalog across time-tracking, day-planning AND month-planning (log once, Sims-style)
**Ask (quote):** *"use those same habits in ur time tracking and day planing and month planing"* — log once, Sims-style (audit 670–673, BIG, asked ×2).

**Buildable?** PARTIAL → the catalog **already** reuses one source everywhere (`allActivities`→`bentoByDomain`→`bentoPicker` feeds tracking via `startTrackerNow`+`assignTimer` `app.js:2254`/`2252`, day-planning via `planSheet`/`makeBlock`, stacks via `presetsSheet`/`stackDetail`, the bento, and onboarding `app.js:1242`). Customs are added **once** to `S.acts` and appear in all of them — the "log once" half is DONE. The genuine gap is **month-planning has no planning surface** — `pullZoom==="month"` (`buildPull` `app.js:860`) shows a read-only `monthGrid` of done/missed dots (`app.js:2550`) you can only tap to *navigate* into a day. **Build: place activities onto future days directly from the month grid, using the same catalog.**

**APPROACH** (two concrete, shippable steps; do Step A first — it's the whole ask's missing primitive)

**Step A — make the month grid a PLANNING surface (the missing piece).**
- In `monthGrid(L, baseK, onDay)` (`app.js:2550`), each `.mocell` currently has `cell.onclick = onDay(dk)` (navigate). Add: a **long-press** (or a tiny corner `＋`) on a *future-or-today* cell → opens `bentoPicker({title:"Plan "+relLabel(dk), onPick:fn})`, where `fn` pushes a `blocks(dk)` entry using the same default-placement helper as the day planner (`{ id:uid(), time:"09:00" or nextFreeTime(dk), mins:30, title:x.title, prio:2, color:x.color, domain:x.domain, catK:x.catK, done:false }`), then `reflow(dk); save();` and re-render the month (`buildPull()`).
  - Use a corner `＋` glyph, NOT only long-press, because long-press on `.mocell` is device-untested and the month view is reached via `buildPull` (zoom). A visible `＋` on hover/tap-hold of future cells is the low-risk path. Tap-center still navigates (`onDay`).
- After adding, bump the cell's dot count immediately so the user sees the plan land (the `modot` logic at `app.js:2558` already colors by done-ratio; a 0-done planned day shows the "miss" mauve `#5e2f49` — acceptable, it just means "planned, none done yet"; consider a neutral "planned" dot color `mixHex(DOM.create.c,"#160510",.4)` when `done===0 && bl.length>0 && dk>=todayK()` so a *future* planned day doesn't read as failed).

**Step B — "apply a stack to a day" from month (Sims-style log-once reuse).**
- `presetsSheet(dk)` (`app.js:538`) + `stackDetail` (`app.js:508`) already let you build a reusable stack of activities and apply it to a day — this IS the Sims-style "compose once, drop many." It's already wired to future days in the day view (`dayHeadInfo` `app.js:870` shows a "stacks" button for `dk>todayK()`). Add the same entry from a month cell's long-press menu: "Plan one…" (Step A bentoPicker) | "Apply a stack…" (`presetsSheet(dk)`). This makes month-planning reuse the exact same catalog + stack system as day-planning — no new catalog, "log once" honored.

**CODE POINTERS**
- `monthGrid(L,baseK,onDay)` `app.js:2550` — add the per-cell `＋`/plan affordance + the future-planned dot color tweak at `app.js:2558`.
- `buildPull` month branch `app.js:860–865` — `monthGrid` is called there with `onDayPick`; pass a richer `onDay` or wire the `＋` inside `monthGrid` directly (preferred — keeps `buildPull` untouched, lower risk).
- Reuse `bentoPicker` `app.js:2709`, `presetsSheet` `app.js:538`, `blocks(dk)` `app.js:1320`, `reflow` (search: `function reflow(`), `nextFreeTime` (used at `app.js:2586`).
- The "log once" reuse already lives in `allActivities`/`S.acts` `app.js:2698`/`2703` — DO NOT rebuild; just point the new month-plan picker at the same `bentoPicker`.

**DATA**
- Writes to existing `blocks(dk)` (future-dated). No new shape. No migration.
- Customs already persist once in `S.acts` and surface in every picker — verified at `allActivities` `app.js:2703`. Nothing to add for "log once."

**REGION:** `monthGrid` (`app.js:2550–2561`) — self-contained, low blast radius. Does NOT touch `calendarView` day-render. Touches `buildPull` only at the read-only month branch if you choose to thread `onDay` (prefer wiring inside `monthGrid` to avoid `buildPull`).

- Picking a place-time: keep it dumb (first free slot / 9am) — don't try to build a full month-scoped time scrubber; David adjusts on the day. The ask is *reuse the catalog to plan ahead*, not a second full planner.
